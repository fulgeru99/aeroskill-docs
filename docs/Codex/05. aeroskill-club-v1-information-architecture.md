# Aeroskill Club V1 Screen-by-Screen Wireframe Specification

Document date: 2026-06-28  
Product: Aeroskill Club web platform  
Version: V1 wireframe planning draft  
Source document: `aeroskill-club-v1-prd.md`

## 1. Purpose

This document translates the V1 product requirements into a buildable screen-by-screen wireframe specification. It defines the visible structure, navigation, primary actions, content zones, states, and data dependencies for the public website, member portal, and admin CRM.

This is not a visual design file. It is a functional layout blueprint for product, design, and development.

## 2. UX Direction

Aeroskill Club should feel like a serious general aviation membership organization: polished, operationally trustworthy, and quietly premium. The product should avoid looking like a generic SaaS landing page or a decorative aviation fan site.

Key experience goals:

- Public visitors should quickly understand why the club exists and how to join.
- Members should always see their membership status, card, renewal date, and benefits without hunting.
- Admins should get dense, scannable CRM screens built for repeated operational use.
- The same core data should appear consistently across public, member, and admin areas.

## 9. Critical Cross-Screen Flows

### 9.1 Public Visitor to Active Member

Flow:

```text
Home
-> Membership
-> Join
-> Register
-> Onboarding Profile
-> Choose Plan
-> Checkout
-> Member Dashboard
-> Member Card
```

UX requirements:

- Selected tier persists across registration and checkout.
- Progress indicator appears during onboarding.
- If payment fails, user returns to checkout with clear recovery.
- After successful activation, user lands on dashboard with card CTA.

### 9.2 Expired Member Renewal

Flow:

```text
Login
-> Member Dashboard
-> Subscription
-> Checkout
-> Dashboard
-> Member Card
```

UX requirements:

- Expired status is visible immediately.
- Benefits are limited or locked according to rules.
- Card validity updates after renewal.

### 9.3 Admin Creates Benefit from Partner

Flow:

```text
Admin Dashboard
-> Organizations
-> Organization Detail
-> Add Contract
-> Contract Detail
-> Add Benefit
-> Benefit Detail
-> Publish
```

UX requirements:

- Organization context carries into contract and benefit forms.
- Benefit preview is available before publishing.
- Eligible tier selection is required.

### 9.4 Contract Renewal Follow-up

Flow:

```text
Admin Dashboard
-> Contracts expiring soon
-> Contract Detail
-> Add Task / Update Status
-> Organization Detail
```

UX requirements:

- Expiring contracts are visible on dashboard.
- Responsible admin is shown.
- Related benefits are visible from contract detail.

## 10. Responsive Behavior

### 10.1 Public Website

Desktop:

- Horizontal navigation.
- Multi-column tier and benefit grids.
- Footer in columns.

Mobile:

- Drawer menu.
- Single-column content.
- Tier cards stacked.
- Sticky or prominent join CTA where appropriate.

### 10.2 Member Portal

Desktop:

- Left navigation.
- Dashboard cards in 2-3 columns.
- Card and benefits visible above fold where possible.

Mobile:

- Top navigation with drawer or bottom nav.
- Member card preview near top.
- One-column layout.
- QR code large enough to scan.

### 10.3 Admin CRM

Desktop:

- Left sidebar.
- Tables with filters.
- Detail screens with tabs.

Tablet/mobile:

- Collapsible sidebar.
- Filters stack above tables.
- Tables may scroll horizontally.
- Critical actions remain visible near page title.

## 11. Empty, Error, and Permission States

### 11.1 Empty States

Every list screen must include:

- Short explanation.
- Primary create action if user has permission.
- Secondary guidance where useful.

Examples:

- No benefits: "Create the first member benefit."
- No organizations: "Add flight schools, sponsors, aerodromes, or other partners."
- No payments: "Payments will appear here after members subscribe."

### 11.2 Error States

Required error patterns:

- Form validation error.
- Payment failure.
- File upload failure.
- Network or server error.
- Permission denied.
- Record not found.

### 11.3 Permission States

If an admin lacks permission:

- Hide create/edit/destructive controls where practical.
- Show read-only views where useful.
- Show a clear restricted message if the whole screen is unavailable.

## 12. Wireframe Acceptance Criteria

The V1 wireframes are implementation-ready when:

- Every P0 public, member, and admin screen has a defined purpose and layout.
- All primary user flows can be completed using the listed screens.
- Every admin module has a list and detail/create/edit pattern.
- Member status, subscription status, and card status are visible where relevant.
- Public, member, and admin navigation models are distinct but coherent.
- Empty, error, and permission states are defined.
- The wireframe spec maps cleanly to the PRD and database schema.
