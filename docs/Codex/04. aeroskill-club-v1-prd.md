# Aeroskill Club V1 Product Requirements Document

Document date: 2026-06-28  
Product: Aeroskill Club web platform  
Version: V1 planning draft  
Strategy: Polished membership club with a strong CRM foundation underneath

## 1. Product Summary

Aeroskill Club V1 is a web platform for a general aviation club. It must help the club present a credible public presence, convert visitors into paying members, issue digital member cards, publish member benefits, and give admins a structured CRM foundation for members, partners, sponsors, flight schools, aerodromes, associations, benefits, contracts, communications, and fleet records.

The first version should not attempt to become a full aviation operations platform. It should be excellent at membership, benefits, basic payments, member recognition, and CRM data hygiene.

## 2. Product Principle

V1 succeeds if Aeroskill Club can:

- Recruit a new member from the public website.
- Collect the member's required profile details.
- Accept or record a membership payment.
- Activate the member's subscription.
- Issue a digital member card with QR validation.
- Show benefits based on membership tier.
- Let admins manage members, partners, benefits, and contracts from one place.

## 3. Target Users

### 3.1 Public Visitors

Public visitors include aviation enthusiasts, student pilots, licensed pilots, aircraft owners, sponsors, flight schools, aerodromes, aviation associations, and potential partners.

Primary needs:

- Understand what Aeroskill Club is.
- Compare membership tiers.
- See practical aviation benefits.
- Trust that the club is legitimate.
- Join or contact the club easily.

### 3.2 Members

Members include enthusiasts, student pilots, pilots, supporters, and aviation professionals who have joined the club.

Primary needs:

- Manage their profile.
- View subscription status.
- Pay or renew membership.
- Access a digital member card.
- View eligible benefits.
- Receive important club communications.

### 3.3 Admins

Admins include club leadership, membership managers, finance users, communications users, and partnership managers.

Primary needs:

- Manage members and subscriptions.
- Track payments and payment status.
- Manage partner organizations and contacts.
- Publish benefits.
- Track contracts and renewals.
- Communicate with members and partners.
- Maintain a basic fleet registry.

## 4. V1 Scope

### 4.1 In Scope

- Public website.
- Membership tier presentation.
- Member registration and login.
- Member onboarding profile.
- Subscription selection.
- Payment recording or payment integration.
- Member dashboard.
- Digital member card.

### 4.2 Out of Scope

- Aircraft booking.

- Social network features.

## 5. Priority Levels

- P0: Required for launch.
- P1: Strong V1 feature; recommended shortly before or after launch.
- P2: Later enhancement; not needed for first launch.

## 6. Membership Tiers

The system must support three public membership tiers in V1. Names and prices should be configurable in admin settings or seed data.

Recommended default tier names:

| Tier | Priority | Target Audience | Notes |
| --- | --- | --- | --- |
| Explorer | P0 | Enthusiasts, newcomers, aviation-curious members | Entry-level membership |
| Pilot | P0 | Student pilots, PPL holders, active GA participants | Core aviation participant tier |
| Aviator Plus | P0 | Owners, frequent flyers, premium supporters | Higher-value tier |

### 6.1 Tier Fields

Each tier must support:

- Name.
- Short description.
- Annual price.
- Currency.
- Public visibility.
- Sort order.
- Benefits summary.
- Renewal period.
- Active/inactive status.

### 6.2 Tier Rules

- A member can have only one active tier at a time.
- A tier determines benefit eligibility.
- Tier names, descriptions, and prices must not be hardcoded into UI copy.
- Admins must be able to deactivate a tier without deleting historical subscription records.

## 7. Roles and Permissions

### 7.1 Public Roles

| Role | Description |
| --- | --- |
| Visitor | Unauthenticated public user |
| Member | Authenticated club member |
| Expired Member | Authenticated member with inactive or expired subscription |

### 7.2 Admin Roles

| Role | Priority | Description |
| --- | --- | --- |
| Super Admin | P0 | Full access to all platform areas |
| Club Admin | P0 | Manage most club operations except high-risk settings |
| Finance Admin | P0 | Manage payments, subscriptions, refunds, invoices |
| Communications Admin | P1 | Manage announcements and templates |
| Partnership Manager | P1 | Manage organizations, contacts, benefits, contracts |
| Fleet Manager | P1 | Manage fleet registry and aircraft documents |
| Read-only Staff | P1 | View CRM records without making changes |

### 7.3 Permission Requirements

- Admin area must be separate from member portal navigation.
- Sensitive actions must require authenticated admin access.
- Payment and subscription changes must be recorded in an audit log once audit logging is implemented.
- Read-only users must not see destructive controls.
- Super Admin must be able to assign roles.

## 12. Core User Flows

### 12.1 Visitor Joins as Member

Priority: P0

Steps:

1. Visitor opens public website.
2. Visitor reviews membership tiers.
3. Visitor chooses tier.
4. Visitor creates account.
5. Visitor completes required profile fields.
6. Visitor confirms terms and required consents.
7. Visitor pays subscription or starts pending payment.
8. System activates subscription after payment confirmation.
9. System issues digital member card.
10. Member lands on dashboard.

Acceptance criteria:

- User can complete signup without admin intervention.
- Admin can see new member in CRM.
- Member card is not active before subscription activation.

### 12.2 Member Renews Subscription

Priority: P0

Steps:

1. Member receives renewal reminder.
2. Member logs in.
3. Member opens subscription status.
4. Member pays renewal.
5. System updates subscription period.
6. System updates member card validity.

Acceptance criteria:

- Expiry date and card validity remain aligned.
- Payment appears in member and admin payment history.

### 12.3 Member Uses Benefit

Priority: P0

Steps:

1. Member logs in.
2. Member opens benefits directory.
3. Member filters by category.
4. Member opens benefit detail.
5. System confirms eligibility based on tier.
6. Member follows redemption instructions.
7. Partner can scan or check card if required.

Acceptance criteria:

- Ineligible members do not see redemption instructions unless the club allows preview.
- Benefit status and validity dates are respected.

### 12.4 Partner Validates Member Card

Priority: P0

Steps:

1. Partner scans member card QR code.
2. Partner opens validation page.
3. System displays card validity.
4. Partner confirms member status.

Acceptance criteria:

- Validation page works without partner login.
- Validation page does not expose sensitive profile details.

### 12.5 Admin Manages Member

Priority: P0

Steps:

1. Admin opens Members.
2. Admin searches or filters.
3. Admin opens member profile.
4. Admin reviews subscription, payments, notes, and card.
5. Admin updates permitted fields.
6. System records important status changes.

### 12.6 Admin Adds Partner and Benefit

Priority: P0

Steps:

1. Admin creates Organization.
2. Admin assigns organization type.
3. Admin adds contact.
4. Admin creates contract.
5. Admin creates benefit linked to organization and contract.
6. Admin assigns eligible tiers.
7. Admin publishes benefit.

Acceptance criteria:

- Benefit appears in member portal only for eligible members.
- Public visibility follows benefit settings.

### 12.7 Admin Tracks Contract Renewal

Priority: P0 for data, P1 for automated reminders

Steps:

1. Admin creates contract.
2. Admin sets end date and renewal reminder date.
3. Dashboard shows expiring contracts.
4. Admin follows up and updates status.

Acceptance criteria:

- Contract can be found from organization page.
- Related benefits can be reviewed from contract page.

## 14. System Rules

### 14.1 Authentication

- Users authenticate with email and password or an external auth provider.
- Admin access requires assigned admin role.
- Password reset must be available.
- Email uniqueness is required.

### 14.2 Membership Activation

- New member profile starts as draft or pending payment.
- Subscription becomes active only after successful payment or manual admin activation.
- Active subscription controls active member card status.

### 14.3 Member Number

- Each member must receive a stable member number.
- Member number must not change when the member renews.
- Member number should be human-readable.

Example format:

- ASC-2026-0001

### 14.4 Member Card

- Card token must be unique and hard to guess.
- Card can be revoked and reissued.
- Card validity must match subscription validity.
- Expired subscription must make card invalid or expired.

### 14.5 Benefit Eligibility

- Benefits can be assigned to one or more tiers.
- Member must have active subscription to access member-only benefits.
- Benefit must be active and within validity dates.
- Admin can pause a benefit without deleting it.

### 14.6 Organization CRM

- Organizations can have multiple types.
- Public partner directory only shows organizations marked public.
- Organizations should not be deleted if linked to contracts or benefits; archive instead.

### 14.7 Contracts

- Contracts are linked to organizations.
- Contracts may be linked to benefits.
- Expired contracts should not automatically remove benefits unless configured by admin.
- Renewal reminder date must be visible in admin dashboard.

### 14.8 Payments

- Payment records must be immutable for financial traceability except limited status updates.
- Refunds must be recorded as status changes or separate refund records depending on implementation.
- Manual payment reconciliation must require admin permission.

### 14.9 Documents

- Documents must have explicit visibility.
- Contract documents are admin-only by default.
- Member-facing documents require explicit member visibility.

### 14.10 Communications

- Service emails are required for account, payment, and subscription operation.
- Marketing or partner-offer communications require explicit preference handling.
- Communications should record audience and send status.

## 15. Non-Functional Requirements

### 15.1 Security

- Passwords must never be stored in plain text.
- Admin routes must be protected.
- Member data must be scoped to the logged-in member.
- Public validation page must expose only approved fields.
- File uploads must restrict allowed file types and sizes.

### 15.2 Privacy

- Required and optional personal data must be clearly separated.
- Members must be able to update communication preferences.
- Privacy policy must reflect the real operating jurisdiction and data processors.
- Legal review is required before launch.

### 15.3 Availability and Reliability

- Public website should remain accessible even if some member services are temporarily unavailable.
- Payment confirmation flow must handle delayed provider webhooks.
- Admins need visibility into failed or pending payments.

### 15.4 Performance

- Public pages should load quickly on mobile connections.
- Admin lists should support search and pagination.
- Benefits and member dashboard should avoid unnecessary heavy media.

### 15.5 Accessibility

- Public and member pages should support keyboard navigation.
- Color contrast must be sufficient.
- Form errors must be clear and associated with fields.
- QR validation page should work on mobile devices.

## 19. Open Questions

### 19.1 Business

- What country or jurisdiction will the club operate under?
- What currency should membership prices use?
- What are the actual prices for the three tiers?
- Is membership annual only, or monthly and annual?
- Are there age restrictions or guardian requirements for younger members?
- Are members required to be pilots, or can anyone join?
- Should Aviator Plus imply voting rights, premium benefits, or sponsorship-style support?

### 19.2 Payments

- Should sponsors have tiered sponsor levels?
- Are contracts legally binding documents stored in the system, or just references?

- Should announcements be sent from a club email domain?
- Are partner promotional emails allowed?
- Who approves communications before sending?

## 21. Implementation Notes

Recommended technical shape:

- Single web application with public, member, and admin areas.
- Relational database with strong entity relationships.
- Role-based access control.
- Payment provider integration with webhook handling.
- Email service for transactional messages.
- Object storage for documents and logos.
- QR card validation via signed or random card token.

Recommended implementation approach:

- Build public and member flows first enough to validate membership conversion.
- Build admin CRM in parallel with simple list/detail/create/edit patterns.
- Keep entity models normalized early.
- Avoid building aviation operations features until membership and CRM are stable.

## 22. Definition of Done for V1

Aeroskill Club V1 is done when:

- Public visitors can understand the club, compare tiers, and join.
- Members can register, complete profile, pay, view subscription, and use a digital card.
- QR validation confirms active membership safely.
- Members can browse eligible benefits.
- Admins can manage members, subscriptions, payments, organizations, contacts, benefits, contracts, and documents.
- Admins have at least basic dashboard visibility into renewals, payments, benefits, partners, and expiring contracts.
- Required system emails work.
- Legal pages are reviewed and published.
- Production environment is configured and tested.
- P0 user flows pass end-to-end testing.
